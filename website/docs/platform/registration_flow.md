# Event Registration Flow Redesign

> **Status: implemented** (T-008–T-010). Live code: Event Hub in [`frontend/src/pages/operations.tsx`](../../frontend/src/pages/operations.tsx) + [`events.go`](../../internal/handlers/events.go). This doc is historical design context.

## Goal
The current registration flow is too simple ("Click register for deployment"). We need to overhaul both the User Experience (UX) and the Backend Architecture to support complex, multi-mission events (like the "Twin Theaters" Star Wars event) where players can browse available missions, view the briefing/assets, choose a faction (side), select a specific squad, and claim a slot.

This plan merges the designs from the local React app, `tbdevent.eu/sw.html` (multi-mission event dossiers), and `register.tbdevent.eu/mission.htm` (faction/squad ORBAT selection).

## Design Principles
1. **Database Refactoring Approved:** The Go backend will be refactored to support multiple sequential missions per event.
2. **Automated ORBATs:** Squads and factions will be automatically generated by the Go backend parsing the uploaded `mission.json` file. No manual squad creation by admins.
3. **Apple-tier Simplicity:** The UX must feel native, clean, and incredibly intuitive—inspired by macOS interface design (e.g., Finder columns, System Settings split-panes).

## Proposed UX Flow

### 1. The Event Hub Page (Replaces basic "Deployments" view)
When a user clicks "Register" on an upcoming Event from the Dashboard:
- They are taken to the **Event Hub** (e.g., `/events/twin-theaters`).
- **Hero Section**: Massive countdown, lore briefing, and universal connection info (TS3 IP, Modpack Download).
- **Mission List**: A stacked list of "Dossiers" (just like `sw.html`).
  - Mission 1: The Pocket (GAR vs CIS)
  - Mission 2: Cold Start (Empire vs Alliance)
- Each Mission dossier shows the briefing, the assets available to each side (e.g., 2x TX-130, 4x AT-ST), and two action buttons:
  - **"Open in Mission Planner"**: A button to open the 2D interactive mission editor map to see the mission in granular detail.
  - **"Open ORBAT"**: Opens the registration flow.

### 2. The ORBAT Selection Interface (Apple macOS / Split-Pane Style)
When the user clicks "Open ORBAT" for a specific mission, they shouldn't get a clunky dropdown form. Instead, they get a sleek, macOS System Settings-style split pane:

- **Left Column (Navigation Sidebar)**: 
  - Top level filters for Factions (e.g., a toggle switch: `[ GAR ] | [ CIS ]`).
  - Below the toggle, a clean vertical list of Squads (e.g., `Platoon HQ`, `Alpha 1-1`, `Air Wing`).
  - Clicking a squad highlights it natively.
- **Right Column (Main Detail Pane)**:
  - Displays the specific slots inside the currently selected Squad from the left column.
  - A clean table/list showing the Role (`Squad Leader`, `Combat Medic`), and Status.
  - Slots are intuitively color-coded: 
    - Green/Empty = Available to claim
    - Filled = Shows the Discord Avatar and name of the player who claimed it.
  - The user simply clicks an empty row and hits a prominent **"Register for Deployment"** button at the bottom of the pane.

## Proposed Backend Architecture Changes

To support this in the Go backend, we need to alter the schemas slightly:

### [MODIFY] `events` table
Remove the strict 1:1 `mission_id` constraint. An event is now a container.
```sql
ALTER TABLE events DROP COLUMN mission_id;
-- Add lore/briefing fields for the overarching event
ALTER TABLE events ADD COLUMN briefing TEXT;
ALTER TABLE events ADD COLUMN banner_image_url TEXT;
```

### [NEW] `event_missions` table
This links an overarching Event to multiple Missions.
```sql
CREATE TABLE event_missions (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    event_id UUID NOT NULL REFERENCES events(id) ON DELETE CASCADE,
    mission_id UUID NOT NULL REFERENCES missions(id),
    start_time TIMESTAMPTZ NOT NULL,
    -- Allows different missions in the same event to run at different times
    UNIQUE(event_id, mission_id)
);
```

### [MODIFY] `orbat_slots` and `event_registrations`
Currently, these link directly to `event_id`. They must be updated to link to `event_mission_id`, so we know *which* mission inside the event the player registered for.
```sql
ALTER TABLE orbat_slots RENAME COLUMN event_id TO event_mission_id;
ALTER TABLE event_registrations RENAME COLUMN event_id TO event_mission_id;
```
